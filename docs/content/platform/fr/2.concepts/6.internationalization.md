TockDocs prend en charge l'internationalisation dans ses deux modes de contenu :

- **mode legacy** — routes préfixées par la locale comme `/en/...` et `/fr/...`
- **mode knowledge base** — routes orientées KB comme `/docs/platform/en/...` et `/docs/platform/fr/...`

## Ce que TockDocs gère pour vous

- les collections de contenu par locale
- la génération des routes localisées
- un sélecteur de langue sur les pages de documentation
- le filtrage des locales sans contenu réel
- le comportement par KB quand différentes knowledge bases exposent des jeux de locales différents

## Sites monolingues

Si votre site est monolingue et n'utilise pas le module complet `@nuxtjs/i18n`, vous pouvez quand même définir la locale UI dans `app.config.ts` :

```ts [app/app.config.ts]
export default defineAppConfig({
  tockdocs: {
    locale: 'fr',
  },
})
```

Cela sert principalement pour les chaînes UI intégrées et les métadonnées de langue du document.

## Configuration multilingue avec `@nuxtjs/i18n`

Pour une vraie documentation localisée, configurez `@nuxtjs/i18n` dans `nuxt.config.ts` :

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  modules: ['@nuxtjs/i18n'],
  i18n: {
    defaultLocale: 'en',
    locales: [
      { code: 'en', name: 'English' },
      { code: 'fr', name: 'Français' },
    ],
  },
})
```

::warning
TockDocs force la stratégie de routage des docs à `prefix` pour garder des URLs localisées explicites.
::

## Structure du contenu en mode legacy

En mode legacy, le contenu est organisé par locale :

```bash
content/
├── en/
│   ├── index.md
│   └── 1.getting-started/
└── fr/
    ├── index.md
    └── 1.getting-started/
```

Routes typiques :

- `content/en/index.md` → `/en`
- `content/fr/1.getting-started/3.installation.md` → `/fr/getting-started/installation`

## Structure du contenu en mode knowledge base

En mode KB, les locales vivent à l'intérieur de chaque knowledge base :

```bash
content/
├── site/
│   └── index.md
└── platform/
    ├── kb.yml
    ├── en/
    │   └── 1.getting-started/
    └── fr/
        └── 1.getting-started/
```

Exemple de `kb.yml` :

```yaml [content/platform/kb.yml]
id: platform
title: Platform
defaultLocale: en
locales:
  - en
  - fr
entry: getting-started/installation
```

Routes typiques :

- `/docs/platform/en/getting-started/installation`
- `/docs/platform/fr/getting-started/installation`

## Filtrage des locales

TockDocs n'enregistre que les locales réellement soutenues par le contenu.

Cela signifie que :

- déclarer une locale dans `i18n.locales` ne suffit pas
- le dossier de contenu correspondant doit aussi exister
- en mode KB, la disponibilité des locales est évaluée par knowledge base

Exemple :

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  modules: ['@nuxtjs/i18n'],
  i18n: {
    defaultLocale: 'en',
    locales: ['en', 'fr', 'ja'],
  },
})
```

Si votre contenu n'existe que pour `en` et `fr`, alors `ja` est ignoré pour le routage documentaire.

## Nombres de locales mixtes en mode KB

Le mode KB prend en charge des nombres de locales différents selon la KB.

Par exemple :

- `platform` peut exposer `en` et `fr`
- `parser` peut n'exposer que `en`

Dans ce cas :

- le sélecteur de langue apparaît sur les KB multilingues
- il reste masqué sur les KB monolingues

## Exigences de locale par défaut

Assurez-vous toujours que la locale par défaut configurée existe réellement dans le contenu :

- en mode legacy : `content/<defaultLocale>/...`
- en mode KB : `content/<kb>/<defaultLocale>/...`

Si une locale docs est absente, TockDocs retombe sur la locale par défaut de la KB ou du site quand c'est possible.

## Starter i18n

Si vous voulez un starter localisé préconfiguré, utilisez :

```bash [Terminal]
npx create-tockdocs my-docs -t i18n
```

Ce starter utilise le mode legacy avec dossiers par locale et une configuration `@nuxtjs/i18n` déjà en place.
