## Migration depuis les anciennes versions de TockDocs

La version actuelle de TockDocs repose sur un **layer Nuxt + le workflow `create-tockdocs`**.

Les principaux changements concernent surtout l'initialisation du projet et les commandes :

| Ancien workflow               | Workflow actuel                    |
| ----------------------------- | ---------------------------------- |
| `npx tockdocs init my-docs`   | `npx create-tockdocs my-docs`      |
| `tockdocs dev`                | `npm run dev`                      |
| `tockdocs build`              | `npm run build`                    |
| `--extends tockdocs` inline   | `nuxt.config.ts` généré étend le layer |

::tip
Votre contenu Markdown existant et la plupart des syntaxes MDC peuvent généralement être migrés sans réécriture majeure. Les principaux changements concernent la structure du projet et les conventions de routage.
::

## Migration depuis n'importe quelle solution Markdown

Que vous veniez d'une ancienne version de TockDocs, du template docs Nuxt UI ou d'un autre site Markdown, le chemin de migration est généralement le suivant :

::steps

### Créez un nouveau projet TockDocs

```bash [Terminal]
npx create-tockdocs my-docs
```

Choisissez :

- `default` pour une documentation monolingue
- `i18n` pour une documentation localisée en mode legacy

### Déplacez votre contenu dans `content/`

Dans la majorité des cas, vous pouvez copier vos fichiers Markdown dans le nouveau dossier `content/`, puis affiner progressivement la structure, la navigation et la syntaxe MDC.

### Choisissez la bonne architecture

Utilisez la structure qui correspond à votre site :

- **mode legacy** pour un seul arbre de documentation
- **mode knowledge base** pour plusieurs KB comme `/docs/platform/en/...` et `/docs/parser/en/...`

### Mettez à jour les routes et les liens

Si votre ancienne documentation utilisait d'autres routes, revérifiez :

- les liens Markdown internes
- les chemins d'images
- les URLs `/docs/...` codées en dur
- les ids de knowledge base et les préfixes de locale

### Revoyez la configuration et les fonctionnalités IA

Une fois le contenu déplacé, relisez les guides [Configuration](/docs/platform/fr/concepts/configuration), [Assistant](/docs/platform/fr/ai/assistant) et [MCP](/docs/platform/fr/ai/mcp) pour réaligner l'application sur les conventions actuelles de TockDocs.
::

## Mode legacy vs mode knowledge base

### Mode legacy

Utilisez-le lorsque votre projet n'a qu'un seul arbre documentaire.

Exemple monolingue :

```bash
content/
├── index.md
└── guide/
    └── introduction.md
```

Exemple localisé :

```bash
content/
├── en/
│   └── guide/
└── fr/
    └── guide/
```

### Mode knowledge base

Utilisez-le lorsque vous voulez plusieurs ensembles de documentation dans un même site :

```bash
content/
├── site/
│   └── index.md
├── platform/
│   ├── kb.yml
│   └── en/
└── parser/
    ├── kb.yml
    └── en/
```

En mode KB, les routes sont générées sous la forme `/docs/<kb>/<locale>/...` et chaque KB possède ses propres collections, métadonnées, locales et page d'entrée.

## Compatibilité du contenu

- **Markdown standard** : peut être copié tel quel, puis enrichi progressivement
- **Nuxt Content + MDC** : se migre souvent bien si les composants référencés existent toujours
- **Composants personnalisés** : peuvent devoir être recréés dans `app/components/`
- **Hypothèses de routage legacy** : peuvent nécessiter une mise à jour lors d'un passage au mode KB

## Checklist pratique

Avant de considérer la migration terminée, vérifiez que :

- le contenu s'affiche sans fuite de marqueurs MDC bruts
- les liens internes pointent vers les nouvelles routes
- les titres et icônes de navigation sont corrects
- les locales sont bien filtrées
- les pages d'entrée des KB fonctionnent correctement avec `kb.yml`
- l'assistant et MCP pointent vers le bon scope documentaire
